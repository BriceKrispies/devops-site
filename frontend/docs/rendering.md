# Rendering Model

## Principle

The browser renders HTML immediately. TypeScript enhances what is already on screen. The user never stares at a blank page.

## Rendering Phases

### Phase 1: Immediate HTML Render

The browser parses and paints the static `.html` file. This produces:

- Full page layout (nav, sidebar, header, content area, footer)
- Section headings and structural containers
- Async regions in their initial `loading` state, with placeholder markup already inline

This phase completes before any JavaScript executes. The HTML file contains everything needed for the first paint.

Example HTML for an async region:

```html
<section data-region="pipelines" data-state="loading">
  <h2>Pipelines</h2>
  <div class="region-loading">
    <div class="skeleton skeleton-row"></div>
    <div class="skeleton skeleton-row"></div>
    <div class="skeleton skeleton-row"></div>
  </div>
</section>
```

The skeleton markup is part of the static HTML. It is visible immediately.

### Phase 2: Progressive Enhancement

The page's entry TypeScript module loads (via `<script type="module">`). It:

1. Creates the store.
2. Registers adapters.
3. Queries all `[data-region]` elements.
4. Binds each region to its store topic(s).
5. Attaches event listeners for interactive elements.

At this point, the page looks identical to Phase 1 — but now the plumbing is active and ready to receive data.

### Phase 3: Async Hydration

Adapters begin publishing data. As each topic receives its first value, the corresponding region transitions from `loading` to its resolved state.

This is **not** framework hydration (attaching event handlers to server-rendered markup). It is simpler: the region's innerHTML is replaced with rendered data content. There is no diffing, no virtual DOM reconciliation.

Each region hydrates independently. A fast adapter resolves its regions immediately; a slow adapter leaves its regions in `loading` state until data arrives.

## Region States

Every `data-region` element cycles through exactly these states:

| State      | `data-state` attribute | Visual                                    |
|------------|------------------------|-------------------------------------------|
| `loading`  | `"loading"`            | Skeleton placeholder (from static HTML)   |
| `resolved` | `"resolved"`           | Rendered data content                     |
| `error`    | `"error"`              | Error message, optional retry action      |
| `empty`    | `"empty"`              | Intentional empty state ("No items yet")  |

State transitions are managed by the region binding in TypeScript:

```ts
function bindRegion(el: HTMLElement, store: Store, topic: string, render: RenderFn) {
  store.subscribe(topic, (payload) => {
    if (payload.status === "error") {
      el.dataset.state = "error";
      el.querySelector(".region-content")!.innerHTML = renderError(payload.error);
    } else if (payload.data.length === 0) {
      el.dataset.state = "empty";
      el.querySelector(".region-content")!.innerHTML = renderEmpty();
    } else {
      el.dataset.state = "resolved";
      el.querySelector(".region-content")!.innerHTML = render(payload.data);
    }
  });
}
```

CSS uses the `data-state` attribute to show/hide the appropriate content:

```css
[data-region][data-state="loading"] .region-content { display: none; }
[data-region][data-state="loading"] .region-loading { display: block; }
[data-region][data-state="resolved"] .region-content { display: block; }
[data-region][data-state="resolved"] .region-loading { display: none; }
/* ... same pattern for error and empty */
```

## Placeholder Design

Placeholders (skeletons) are static HTML elements styled to approximate the shape of the resolved content. They are not generated by JavaScript.

Guidelines:

- Use `skeleton-row` for list items (a gray bar the height of a text line).
- Use `skeleton-card` for card-shaped content.
- Use `skeleton-chart` for chart-shaped regions.
- Match the approximate dimensions and count of the expected content. Three skeleton rows for a list that will typically show 3–5 items.
- Placeholders should pulse or animate subtly (CSS animation, no JS) to signal that content is loading.

## Error States

When a region enters `error` state:

- Display a short, human-readable error message.
- Optionally display a "Retry" button that re-triggers the adapter for that topic.
- Do not show stack traces, raw error objects, or technical details in the UI.

## Empty States

When a region enters `empty` state (data loaded successfully but there are zero items):

- Display a clear message ("No pipelines configured", "No active alerts").
- Optionally display a call-to-action if appropriate ("Create your first pipeline").
- Do not show the skeleton placeholder. Empty is a resolved state, not a loading state.

## Non-Blocking Rendering Rules

1. **Scripts are deferred.** Use `<script type="module">` which is deferred by default, or place scripts at the end of `<body>`. Never use synchronous `<script>` tags in `<head>` that block parsing.

2. **No synchronous data fetching.** No adapter may call a blocking operation during initialization. All data fetching is async.

3. **DOM writes are batched where possible.** When updating a region, set `innerHTML` once — do not make multiple sequential DOM mutations.

4. **Heavy rendering uses idle scheduling.** If rendering a large dataset (100+ items), use `requestIdleCallback` or chunk the work into smaller batches to avoid long tasks.

5. **Images and heavy assets are lazy-loaded.** Use `loading="lazy"` on images. Do not load off-screen assets eagerly.
